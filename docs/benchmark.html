PERFORMANCE BENCHMARK

Run the actual C++ matching engine and measure throughput and latency. In WASM mode, timing resolution is ~0.1 ms (browser limitation) — latencies are per-operation averages over batches of 10,000 (~10ns precision). In NATIVE C++ mode (run ./engine_server), timing uses std::chrono::high_resolution_clock for true nanosecond resolution.

▶ HOW TO ENABLE NATIVE C++ MODE
  1. Download the binary for your OS from GitHub Releases:
    engine_server-mac-arm64  ·  engine_server-mac-x86_64  ·  engine_server-linux-x86_64  ·  engine_server-windows-x64.exe
  2. Run it in a terminal:
    macOS / Linux: chmod +x engine_server-mac-arm64 && ./engine_server-mac-arm64
    Windows: double-click or run engine_server-windows-x64.exe in PowerShell
  3. Reload this page. The mode badge will switch to ● NATIVE C++ automatically.

What to expect: 12–19M ops/sec (standard mix) · up to 53M ops/sec cancel-heavy · p50 47–85 ns · p99 ≤260 ns · true nanosecond resolution via std::chrono::high_resolution_clock. The server runs locally — no data leaves your machine.
Safari on GitHub Pages: Safari blocks localhost connections from HTTPS pages. Use Chrome or Firefox, or serve the page locally via python3 -m http.server 8080.

WASM: timing uses performance.now() (~0.1 ms resolution); latencies shown are per-op averages across 10,000-op batches — accurate to ~10 ns per op. Native C++: true nanosecond resolution via std::chrono::high_resolution_clock. Throughput peaks at lower op counts (~1–5M) when the order-ID map fits in L3 cache; at higher counts the growing book causes cache pressure and throughput decreases — this reflects real-world engine behavior under sustained load.
